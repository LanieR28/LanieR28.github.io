技術解説

CHUNITHM 雲に凪 Bot

見た目の派手さだけではなく、 複数の処理系をひとつの体験に束ねる設計力です。

このページでは、Bot の機能紹介ではなく、 どう分割し、どう統合し、どう運用に乗せたかを中心にまとめています。 コマンド処理、画像レンダリング、データ同期、キャッシュ戦略、運用保守をひとつの制作物として扱っています。

構成を見る 見せ方の軸へ
構成 6 つの制作単位

メイン Bot と複数レンダラ、同期処理を役割ごとに分割しています。

I/O Bot → 処理 → 画像

入力解釈から画像生成・返信までを一気通貫で設計しています。

同期 公式 + 手動入力

公式サイト由来の記録と手動追加入力を同一モデルに解決しています。

運用 キャッシュ + デプロイ

重い描画処理とサーバ配備を前提に、再生成と静的資産を整理しています。

システム構成

1つの Bot ではなく、
複数レイヤーの連携として設計。

01

入力解釈

QQ から受けたコマンドを解釈し、別名や難易度補完、品質指定までを整理します。

02

処理分配

Bot 本体は処理の入口に徹し、B30・楽曲情報・レベル一覧・譜面表示などを専用系に振り分けます。

03

描画実行

複数のレンダリング系で UI 画像を生成し、PNG 品質や送信サイズ制御まで含めて整えます。

04

返信・運用

生成物を Bot 返信に戻し、キャッシュ・送信品質・失敗時の保守も含めてシステム化しています。

実装で見せられる点

このページで見せたいのは、
“機能数” ではなく “実装の幅”。

コマンド解析と正規化

曲名の別名、色プレフィックスによる難易度推定、'master' / 'ultima' などの表記ゆれを吸収し、 人が雑に入力しても使える入口を作っています。

画像生成と視覚設計

単に値を返すのではなく、B30・chuinfo・レベル一覧・ボーダー分析を 画像 UI として成立させるためのレイアウト、品質、出力サイズ制御を実装しています。

データ統合

公式サイトからの同期結果と、手動入力や追加入力を矛盾なく解決し、 同一プレイヤーの記録として扱えるようにしています。

運用と保守

サーバ同期、Bot 切替、画像キャッシュ、品質プリセット、エラーハンドリングなど、 実際に運用し続ける前提で調整しています。

機能別の証拠

見せ方ではなく、
どの技術を実際に束ねているか。

表示系

B30 系 4 種

Best 30 / AJ / AJC / PC という同一記録の異なる切り方を、 それぞれ別用途のビューとして実装しています。

可視化系

楽曲・譜面・レベル一覧

song info、譜面表示、全曲レベル一覧、個人進行状況という複数の視点を それぞれ別 UI として成立させています。

解析系

ボーダー計算の画像化

数字の計算結果だけではなく、目標達成に必要な余白を視覚として返し、 実用性と読みやすさの両方を確保しています。

運用設計

作って終わりではなく、
継続運用できる構造にする。

静的キャッシュ

重い画像や静的譜面を事前生成し、再描画コストと返信速度の両立を図っています。

品質プリセット

標準・高品質・超高品質の送信サイズを分け、見た目と実運用の妥協点をコントロールしています。

同期パイプライン

公式サイト由来のデータ取得からサーバ反映までを、手順化して壊れにくくまとめています。

複数 Bot 経路

QQ 公式 Bot と NapCat の両系統を扱い、表現力と安定性のバランスを見ながら運用しています。

この作品で見せたいこと

この作品は、
“何が作れるか” より “どう設計できるか” を見せやすい。

1

UI を作る力

見た目の整った画像 UI を、単発のデザインではなく運用可能な出力系として作っています。

2

システムを分ける力

Bot、描画、同期、データ解決を別責務として分割し、全体としてつなぎ直しています。

3

現実に耐える力

入力の揺れ、画像サイズ、キャッシュ、同期失敗、公開運用といった実運用の問題に向き合っています。
