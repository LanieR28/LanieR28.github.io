技術解説

CHUNITHM 雲に凪 Bot

見た目の派手さだけではなく、 複数の処理系をひとつの体験に束ねる設計力です。

このページでは、Bot の機能紹介ではなく、 どう分割し、どう統合し、どう運用に乗せたかを中心にまとめています。 コマンド処理、画像レンダリング、データ同期、キャッシュ戦略、運用保守をひとつの制作物として扱っています。

構成を見る 見せ方の軸へ
構成 6 つの制作単位

メイン Bot と複数レンダラ、同期処理を役割ごとに分割しています。

I/O Bot → 処理 → 画像

入力解釈から画像生成・返信までを一気通貫で設計しています。

同期 公式 + 手動入力

公式サイト由来の記録と手動追加入力を同一モデルに解決しています。

運用 キャッシュ + デプロイ

重い描画処理とサーバ配備を前提に、再生成と静的資産を整理しています。

調査と立ち上げ

最初に決めたのは、
“どの体験を 1 つに束ねるか”。

出発点

最初は単発の画像生成やスコア表示から始まりましたが、使っていくうちに 「B30」「楽曲情報」「レベル別の進行確認」「ボーダー計算」「同期」が ばらばらに存在していること自体が使いにくさだと分かりました。

調査の軸

先に画面を作るのではなく、実際にどの場面で使うかを整理しました。 何をすぐ確認したいか、どこが手入力だとつらいか、どの画面は静的にしてよいかを 先に切り分けてから設計しています。

立項の判断

その結果、単なるソーシャルメディア向け Bot ではなく、 「入力を受ける入口」「画像として返す出力」「データを同期する裏側」を まとめて 1 つのシステムとして扱う方針に変えました。

見せたい価値

この作品で見せたいのは、機能を足したことではなく、 必要な処理を責務ごとに分けて、最後に 1 つの操作体験として再構成した点です。

システム構成

1つの Bot ではなく、
複数レイヤーの連携として設計。

01

入力解釈

QQ から受けたコマンドを解釈し、別名や難易度補完、品質指定までを整理します。

02

処理分配

Bot 本体は処理の入口に徹し、B30・楽曲情報・レベル一覧・譜面表示などを専用系に振り分けます。

03

描画実行

複数のレンダリング系で UI 画像を生成し、PNG 品質や送信サイズ制御まで含めて整えます。

04

返信・運用

生成物を Bot 返信に戻し、キャッシュ・送信品質・失敗時の保守も含めてシステム化しています。

データ収集と整形

一番手間がかかったのは、
“画像” ではなく “入力データ” の扱い。

公式サイト同期

CHUNITHM-NET から記録を取得する経路を整え、 成績、プレイ回数、プロフィール情報を Bot 側で再利用できる形式に変換しています。 単に取得するだけでなく、あとで描画系が扱いやすい形に落とすことを重視しました。

手動入力との統合

公式データだけでは埋まらないケースがあるため、 手動アップロードや PC 入力も同じプレイヤーモデルに流し込んでいます。 同点時の優先順位や、公式由来と手動由来の扱いも整理しています。

別名と入力揺れ

曲名の別名、短縮入力、難易度略称、色プレフィックスなど、 人間側の入力の雑さを吸収する処理を先に挟むことで、 Bot が現実の使われ方に耐えるようにしています。

静的資産の整備

譜面プレビューやレベル一覧のような重い表示は、 その場で毎回ゼロから作るのではなく、静的に持てる部分を切り出して再利用しています。

実装で見せられる点

このページで見せたいのは、
“機能数” ではなく “実装の幅”。

コマンド解析と正規化

曲名の別名、色プレフィックスによる難易度推定、'master' / 'ultima' などの表記ゆれを吸収し、 人が雑に入力しても使える入口を作っています。

画像生成と視覚設計

単に値を返すのではなく、B30・chuinfo・レベル一覧・ボーダー分析を 画像 UI として成立させるためのレイアウト、品質、出力サイズ制御を実装しています。

データ統合

公式サイトからの同期結果と、手動入力や追加入力を矛盾なく解決し、 同一プレイヤーの記録として扱えるようにしています。

運用と保守

サーバ同期、Bot 切替、画像キャッシュ、品質プリセット、エラーハンドリングなど、 実際に運用し続ける前提で調整しています。

途中でぶつかった問題

作った機能より、
途中で壊れた部分の方が設計を育てた。

画像サイズと返信速度

画像を綺麗にしすぎると送信が遅くなり、Bot らしい体験を壊します。 そのため、通常品質・高品質・超高品質を分け、 特に譜面画像だけは別扱いにするなど、用途別の出力戦略を作りました。

プラットフォーム差

QQ 公式 Bot と NapCat では、同じ画像送信でも制限や挙動が異なります。 その差を実地で確認しながら、送信方式と返し方を切り替えられるようにしています。

重い描画処理

特にレベル一覧や複数分割画像は、描画時間そのものが長くなりやすく、 単純な再計算では待ち時間が大きくなります。 そのため静的底図・動的差分・出力キャッシュに寄せる方向で整理しました。

運用中のバグ修正

別名認識、難易度解釈、誤爆しやすい部分一致、送信失敗時の再試行など、 実際に使ってもらって初めて見える問題を継続的に回収しています。

機能別の証拠

見せ方ではなく、
どの技術を実際に束ねているか。

表示系

B30 系 4 種

Best 30 / AJ / AJC / PC という同一記録の異なる切り方を、 それぞれ別用途のビューとして実装しています。

可視化系

楽曲・譜面・レベル一覧

song info、譜面表示、全曲レベル一覧、個人進行状況という複数の視点を それぞれ別 UI として成立させています。

解析系

ボーダー計算の画像化

数字の計算結果だけではなく、目標達成に必要な余白を視覚として返し、 実用性と読みやすさの両方を確保しています。

運用設計

サーバに載せて初めて見える、
公開運用の問題まで含めて設計する。

静的キャッシュ

重い画像や静的譜面を事前生成し、再描画コストと返信速度の両立を図っています。

品質プリセット

標準・高品質・超高品質の送信サイズを分け、見た目と実運用の妥協点をコントロールしています。

同期パイプライン

公式サイト由来のデータ取得からサーバ反映までを、手順化して壊れにくくまとめています。

複数 Bot 経路

QQ 公式 Bot と NapCat の両系統を扱い、表現力と安定性のバランスを見ながら運用しています。

サーバ環境の整理

同期用サービス、Bot 本体、補助的な配備スクリプトを分けて運用し、 画像資産や CSV データも含めて、どこまでを同期し、どこからを再生成にするかを決めています。

落ちた時の復旧

Bot 側の接続失敗、同期失敗、サービス再起動、ログ確認など、 壊れた時にどこを見ればよいかまで整理して、 作ったあとも保守できる状態を目指しています。

この作品で見せたいこと

この作品は、
“何ができるか” より “どう作り上げたか” を見せやすい。

1

UI を作る力

見た目の整った画像 UI を、単発のデザインではなく運用可能な出力系として作っています。

2

システムを分ける力

Bot、描画、同期、データ解決を別責務として分割し、全体としてつなぎ直しています。

3

現実に耐える力

入力の揺れ、画像サイズ、キャッシュ、同期失敗、公開運用といった実運用の問題に向き合っています。
